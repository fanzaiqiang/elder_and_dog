##### 2.1 解決節點過多導致的崩潰

**問題現象：**
```
[ERROR] [rcl]: Failed to find a free participant index
[ERROR] All ROS2 nodes crash with red text
```

**問題診斷：**
- SLAM Toolbox + Nav2 啟動時需要 20+ 個 ROS2 節點
- CycloneDDS 預設 `MaxAutoParticipantIndex` 只有 10-20
- 節點數量超過上限，導致全體崩潰

**解決方案：**
```xml
<!-- Mac VM: ~/ros2_ws/src/elder_and_dog/go2_robot_sdk/config/cyclonedds.xml -->
<Discovery>
    <ParticipantIndex>auto</ParticipantIndex>
    <MaxAutoParticipantIndex>120</MaxAutoParticipantIndex>
    <SPDPInterval>1000ms</SPDPInterval>
</Discovery>
```

**驗證結果：**
- ✅ `robot.launch.py` 成功啟動所有節點
- ✅ SLAM Toolbox、Nav2、go2_driver_node 全部正常運行
- ✅ 無任何紅字錯誤

#### 阻塞點：Ubuntu VM 無法 Ping 到機器狗

**問題現象：**
- Mac 主機能 Ping 到機器狗 ✅ (192.168.12.117 → 192.168.12.1)
- Ubuntu VM 無法 Ping 到機器狗 ❌ (192.168.64.2 → 192.168.12.1, 100% packet loss)

**網路拓樸分析：**
```
機器狗 Go2 (192.168.12.1)
  ↑
  | Wi-Fi (192.168.12.0/24)
  |
Mac 主機 (192.168.12.117)
  ↑
  | UTM NAT (192.168.64.0/24)
  |
Ubuntu VM (192.168.64.2)
```

**問題根因：** VM 與機器狗不在同一網段，需要路由轉送

**嘗試的解決方案（未成功）：**
1. VM 內加入靜態路由 (`sudo ip route add 192.168.12.0/24 via 192.168.64.1`) - ❌ 失敗
2. Mac 主機啟用 IP 轉送 (`sudo sysctl -w net.inet.ip.forwarding=1`) - ❌ 失敗
3. 檢查 Mac 防火牆設定 - ❌ 非防火牆問題

**詳細症狀：**
- Mac 主機已開啟 `net.inet.ip.forwarding=1`
- VM 已設定 `ip route` 指向 Mac
- 封包可以從 VM 送出 (Outbound)，但機器狗收到封包時，來源 IP 是 VM 的內網 IP (`192.168.64.2`)

**根本原因：**
- 機器狗的 Gateway 不知道 `192.168.64.0/24` 網段該往哪送，導致封包有去無回
- 需在 Mac 端做 **NAT Masquerading (IP 偽裝)** 或改用 **Bridged Mode**

## 📊 技術收穫

### 1. CycloneDDS MaxAutoParticipantIndex 參數

**關鍵發現：**
- `MaxAutoParticipantIndex` 決定了同時運行的 ROS2 節點數量上限
- 預設值通常只有 10-20，對於複雜系統（Nav2 + SLAM）不夠
- 建議設定為 120 以上

**計算公式：**
```
所需數量 = 基礎節點 + SLAM 節點 + Nav2 節點 + 自定義節點 + 緩衝
         = 5 + 10 + 15 + 5 + 85 (緩衝) = 120
```

---

### 2. CycloneDDS 版本兼容性

**兼容性對照表：**

| XML 標籤 | 舊版支援 | 新版支援 | 建議 |
|---------|---------|---------|------|
| `<NetworkInterfaceAddress>IP</NetworkInterfaceAddress>` | ✅ | ✅ | 使用 (兼容性好) |
| `<NetworkInterface>eth0</NetworkInterface>` | ❌ | ✅ | 避免 (舊版不支援) |
| `<AllowMulticast>true</AllowMulticast>` | ❌ | ✅ | 避免 (舊版不支援) |

**教訓：** 跨平台開發時，使用最保守的 XML 標籤以確保兼容性

---

### 3. Zsh vs Bash 的跨 Shell 陷阱

**問題發現：**
- 在 Mac VM 設定 ROS2 環境變數時，使用 `source setup.bash` 在 zsh 環境下導致路徑解析錯誤
- 症狀：某些環境變數未正確載入，導致 `ros2` 指令找不到套件

**解決方案：**
```bash
# 錯誤做法 (在 zsh 環境)
source install/setup.bash

# 正確做法 (在 zsh 環境)
source install/setup.zsh
```

**教訓：**
- 不同 Shell 對環境變數與路徑的處理方式不同
- ROS2 提供了多種 Shell 的 setup 腳本 (`setup.bash`, `setup.zsh`, `setup.sh`)
- **最佳實踐**：檢查當前 Shell (`echo $SHELL`)，使用對應的 setup 腳本

### 4. UTM 虛擬機網路架構

**UTM 網路模式對比：**

| 模式 | 優點 | 缺點 |
|------|------|------|
| Shared Network (NAT) | 簡單，VM 自動上網 | VM 與外部設備不在同網段 |
| Bridged (橋接) | VM 直接取得實體 IP | 需手動配置 IP |
| Host-only | 安全，僅 Host ↔ VM | VM 無法上網 |

**當前問題：** 使用 NAT 模式，VM 與機器狗不在同網段

**解決方案（待實施）：**
1. Plan A：改用 Bridged Mode
2. Plan B：使用 rosbag 錄製數據
3. Plan C：在 Mac 主機上直接運行 ROS2
