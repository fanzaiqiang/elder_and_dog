## 📌 今日問題：影像頻率極低

- `/camera/image_raw` 僅 **~2 Hz**，即便有線 WebRTC 連線成功。
- 帶寬壓縮（image_transport）後，頻率仍無改善。

## 🔍 根因判斷

- **瓶頸不在網路頻寬**，而是 **VM 內部 Raw 圖搬運與 CPU 解碼負載**。
- Raw 影像單張約 **2.76 MB**，在 ROS2 內部傳遞成本高，導致整體 FPS 被拖垮。

## ✅ 解法：驅動層直接發布 CompressedImage

### 核心策略
- **關閉 Raw Image 發布（預設）**
- **驅動層直接輸出 JPEG** (`/camera/image_raw/compressed`)
- **避免 image_transport 轉發**，減少多次記憶體搬運

### 新增參數（go2_driver_node）
- `publish_raw_image`：預設 `False`
- `publish_compressed_image`：預設 `True`
- `jpeg_quality`：預設 `80`

## 📊 成果數據

- **頻率**：`/camera/image_raw/compressed` 約 **13–15 Hz**
- **單張大小**：~**0.12 MB**
- **帶寬**：~**1.6–1.9 MB/s**

> 相較原本 2 Hz / 2.76 MB，已提升為可用等級。

---

## 🧪 驗證流程

```bash
ros2 topic list | grep -E "camera|compressed"
ros2 topic hz /camera/image_raw/compressed
ros2 topic bw /camera/image_raw/compressed
```

> 注意需移除舊的壓縮節點（例如 `image_compressor`），避免 publisher 重複。

---

## ⚠️ 觀察事項

- 若 `ros2 topic info /camera/image_raw/compressed` 出現 **2 個 publisher**，代表仍有殘留壓縮節點。
- 需確認 build + source 後再啟動，否則仍會跑舊版程式。

驅動層直接輸出壓縮影像是當前最佳策略，可穩定支撐 Foxglove 即時視覺化。
